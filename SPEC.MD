# Описание приложения

## Подход к проектирвоанию
- API First - сначала фиксируем поведение системы и API-контракт, и только потом переходим к реализации (TypeSpec)

## Подход к разработке
- TDD - Разработка через тестирование
    - Red (Красный): Написание небольшого теста, описывающего желаемое поведение, который сразу падает, так как функционал еще не создан
    - Green (Зеленый): Написание минимально необходимого объема кода, чтобы тест прошел
    - Refactor (Рефакторинг): Улучшение структуры кода (читаемость, оптимизация) при уверенности, что он работает корректно благодаря тесту

## Git Workflow

- Commit after each task completion
- Commit message format: feat: <description> or fix: <description>
- Never commit node_modules/, .next/, tsp-output/, vendor/, .env
- Run linter/tests before committing

## В проекте есть две роли
- владелец календаря
- гости календаря

## В этом проекте нет регистрации и авторизации
- Владелец календаря — один заранее заданный профиль
- Гость бронирует слоты без создания аккаунта и без входа в систему

## Владелец календаря может
- Создавать типы событий. Для каждого типа события задает id, название, описание и длительность в минутах
- Просматривает страницу предстоящих встреч, где в одном списке показаны бронирования всех типов событий

## Гость
- Может посмотреть страницу с видами брони, где доступно название, описание и длительность
- Выбирает тип события, открывает календарь и выбирает свободный слот
- Создает бронирование на выбранный слот

## Правило занятости
- На одно и то же время нельзя создать две записи, даже если это разные типы событий

# Стэк
- TypeSpec — инструмент, в котором мы описываем API-контракт и при необходимости генерируем из него OpenAPI-спецификацию
- backend: Go
- frontend: TypeScript + Vite + React SPA
- UI: shadcn/ui или Mantine (на выбор разработчика)
- API mocking: Prism (для эмуляции API по контракту во время разработки)
- db: PostgreSQL
- разворачивается в docker compose на стенд по ssh

## Backend

### Рекомендуемые технологии
- **TDD**
- **Go** — основной язык бэкенда
- **PostgreSQL** — база данных
- **Docker** — контейнеризация и запуск БД
- **Docker Compose** — оркестрация сервисов

### Требования
- База данных должна запускаться в Docker
- Реализация идет от контракта (TypeSpec), а не от фреймворка

### Задачи бэкенда
1. Реализовать API по созданной TypeSpec-спецификации на Go
2. Реализовать обработку ситуации, когда выбранный слот уже занят
3. Обеспечить хранение данных в PostgreSQL

### Результат
Рабочий бэкенд с хранением данных, соответствующий API-контракту

## Фронтенд

### Рекомендуемые технологии
- **TDD + Playwright**
- **TypeScript + Vite** — быстрая сборка и разработка
- **shadcn/ui** или **Mantine** — UI-компоненты
- **Prism** — эмуляция API по контракту для разработки и проверки

### Преимущества стека
- ИИ-ассистенты дают точные подсказки для этих технологий
- Проще запускать проект
- Безопаснее работать с типами
- Легче вносить изменения по мере развития интерфейса

### Задачи фронтенда
1. Выбрать стек для фронтенда
2. Реализовать страницы интерфейса
3. Подключить интерфейс к API по контракту

### Результат
Появился рабочий UI приложения, следующий контракту API

# Задачи
- Опишите доменные сущности: владелец, тип события, слот, бронирование и публичный сценарий гостя.
- Сформулируйте задачу агенту и подготовьте TypeSpec-спецификацию, которая фиксирует API-контракт.
- Проверьте, что спецификация покрывает сценарии владельца и гостя, а также правило занятости слота.
- TypeSpec-спецификация должна содержать выалидацию полей (email format, duration > 0)
- Должны быть описаны HTTP статус-коды в OpenAPI
- Необходима пагнинация для списка бронирвоаний
- Необходима автогенерация слотов с интервалом 15 или 30 минут (настраивается)
- Необходимо подключить linter TypeSpec и добавить для него команду запуска через Makefile
- Сайт должен поддерживать выбор языка между русским и английским
- Сайт должен поддерживать выбор темы и использовать по умолчанию цветовую тему пользователя
- Необходимы предупреждения о куки согласно локальному законодательству
